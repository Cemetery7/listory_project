# PAGES.md

# Листория — Pages Specification

> Документ описывает каждую страницу проекта, её назначение, структуру, компоненты и пользовательские сценарии.

---

# 1. Общие правила

Каждая страница обязана:

- использовать AppShell;
- использовать общие компоненты;
- использовать Design Tokens;
- поддерживать Dark Theme;
- поддерживать Light Theme;
- быть полностью адаптивной;
- иметь Loading State;
- иметь Empty State;
- иметь Error State.

---

# 2. Общий Layout

Все страницы используют одинаковую структуру.

```text
AppShell

├── Header
├── Sidebar
├── Main Content
└── Right Panel
```

На мобильных устройствах:

```text
Drawer

↓

Header

↓

Main

↓

Widgets
```

# 3. Главная страница

## Route

```text
/
```

---

## Назначение

Первый экран сайта.

Он должен:

- заинтересовать пользователя;
- показать популярные произведения;
- дать возможность быстро начать читать.

---

## Основные задачи

Пользователь должен понимать:

- что это за сервис;
- что здесь можно читать;
- что здесь можно публиковать;
- где искать произведения.

# 30. Global Navigation

Все страницы проекта должны быть связаны между собой.

Основные маршруты:

```text
/

/catalog

/search

/collections

/write

/profile

/settings

/about
```

Ни одна страница не должна быть тупиком.

Пользователь всегда должен иметь возможность:

- вернуться назад;
- перейти на главную;
- перейти в каталог;
- открыть профиль.

# 35. Error Rules

Если загрузка страницы завершилась ошибкой:

Показать:

- понятный заголовок;
- краткое описание;
- кнопку повторной попытки.

Никогда не показывать пользователю технические ошибки.

Например:

❌

```text
TypeError
```

или

```text
500 Internal Server Error
```

Показывать нельзя.

# 40. Codex Page Rules

Перед созданием новой страницы Codex обязан:

1. Прочитать `PROJECT.md`.
2. Прочитать `DESIGN_SYSTEM.md`.
3. Прочитать `COMPONENTS.md`.
4. Прочитать `PAGES.md`.
5. Проверить, существует ли необходимый компонент.
6. Использовать только готовые UI-компоненты.
7. Не создавать новые стили внутри страницы.
8. Не хранить mock data в `page.tsx`.
9. Использовать AppShell.
10. Соблюдать маршрутизацию проекта.
