# PROJECT.md

# Листория

**Листория** — современная веб-платформа для чтения и публикации художественных произведений.

Проект вдохновлен UX-паттернами литературных платформ, но не копирует их дизайн, код, тексты или структуру. Главная цель — создать красивый MVP для хакатона с качественным интерфейсом, понятной архитектурой и AI-функциями.

---

# 1. Главная цель проекта

Создать веб-приложение, которое выглядит как готовый современный продукт.

Приоритеты:

1. Качественный внешний вид.
2. Удобная главная страница.
3. Каталог произведений.
4. Страница произведения.
5. Режим чтения.
6. Создание произведения.
7. Заготовка под AI-функции.
8. Простая архитектура, которую можно развивать дальше.

---

# 2. Продуктовая идея

Пользователь может:

- читать произведения;
- искать истории по жанрам, тегам, категориям и фандомам;
- смотреть популярные произведения;
- открывать новинки;
- сохранять произведения в библиотеку;
- создавать собственные истории;
- получать AI-помощь при публикации.

Основная идея:

> Читать, писать и находить истории должно быть так же приятно, как пользоваться современным SaaS-продуктом.

Референсы:

# 4. Главные страницы MVP

В MVP нужно реализовать:

1. Главная страница `/`

# 5. Главная страница

Главная страница — самый важный экран MVP.

Она должна содержать:

- левый Sidebar;
- верхний Header;
- большой Hero Banner;
- блок “Популярное”;
- блок “Новинки”;
- блок “Продолжаются”;
- блок “Завершённые”;
- блок “Подборки”;
- правую колонку:
  - популярные авторы или фандомы;
  - категории;
  - популярные теги;
  - последние обновления;
  - CTA-блок для входа или публикации.

Главная должна выглядеть как законченный продукт, даже если данные пока mock.

# 6. Основные сущности

В проекте есть следующие сущности:

Пользователь платформы.

Поля:

## Work

Произведение.

- slug
- description
- cover
- authorId
- categoryId
- status
- rating

- likes
- commentsCount

- updatedAt

Глава произведения.

Поля:

- id

Категория произведения.

Примеры:

- Оригинальные произведения

- Поэзия
- Мини
- Макси
- Драма
- Романтика

Тег произведения.

Примеры:

- драма
- романтика
- приключения
- фэнтези
- hurt/comfort
- повседневность

## Fandom

Фандом или мир произведения.

Для mock-данных не использовать реальные защищенные названия.

## Comment

Комментарий к произведению или главе.

## Bookmark

Закладка пользователя.

## Like

Лайк произведения.

## Collection

Подборка произведений.

# 8. Что НЕ делать в MVP

Не тратить время на:

- личные сообщения;
- форум;
- донаты;
- сложную модерацию;
- админ-панель;
- мобильное приложение;
- достижения;
- сложные уведомления;
- систему подписок;
- WebSocket.

# 13. Правила дизайна

Codex обязан соблюдать:

- одинаковые отступы;
- одинаковые border-radius;
- единые цвета через CSS variables;
- поддержку dark/light themes;
- отсутствие случайных цветов;
- отсутствие inline styles;
- отсутствие хаотичных размеров;
- адаптивность;
- читаемость текста.

Все цвета, размеры, тени и анимации будут описаны в `DESIGN_SYSTEM.md`.

---

# 14. Правила кода

Запрещено:

- использовать `any` без необходимости;
- создавать огромные компоненты;
- смешивать UI и бизнес-логику;
- дублировать mock data;
- писать стили inline;
- использовать случайные HEX-цвета внутри компонентов;
- ломать существующую структуру проекта;
- писать весь код в `page.tsx`.

Обязательно:

- TypeScript types;
- маленькие компоненты;
- переиспользуемые UI-блоки;
- чистая структура;
- понятные имена файлов;
- понятные props;
- responsive-first подход;
- accessibility.

# 16. Инструкции для Codex

Перед началом работы Codex должен:

1. Прочитать `docs/PROJECT.md`.
2. Прочитать `docs/DESIGN_SYSTEM.md`, если он уже существует.
3. Не начинать backend, пока frontend MVP не готов.
4. Сначала объяснить план.
5. Затем перечислить файлы, которые будут созданы.
6. После этого писать код.

После каждого этапа Codex должен:

- перечислить измененные файлы;
- объяснить, что реализовано;
- указать следующий шаг;
- проверить, что проект собирается.

# 17. Definition of Done

Этап считается завершенным, если:

- проект запускается;
- TypeScript не выдает ошибок;
- нет очевидного дублирования;
- компоненты переиспользуемые;
- дизайн соответствует референсам;
- работает dark/light theme;
- интерфейс адаптивен;
- нет незавершенных TODO;
- UI выглядит как готовый продукт.

---

# 18. Главный принцип

Если есть выбор между сложной функцией и красивым законченным интерфейсом, выбирать интерфейс.

Для хакатона важнее показать:

- вау-эффект;
- понятный сценарий;
- аккуратный UI;
- AI-фичу;
- целостность продукта.

---

# 19. Демо-сценарий

На защите нужно показать:

1. Пользователь открывает главную.
2. Переключает светлую/темную тему.
3. Открывает каталог.
4. Фильтрует произведения.
5. Открывает страницу произведения.
6. Запускает AI Summary.
7. Открывает режим чтения.
8. Создает новое произведение.
9. AI предлагает теги и описание.
10. Произведение появляется в интерфейсе.
